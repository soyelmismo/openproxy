# Log Detail Modal Refactor — Frontend Spec

## Overview

Refactor the **Response** and **Request** tabs of the admin dashboard's "Live Logs" detail modal to render their content as friendly, collapsible sections instead of a single raw-JSON dump. The four-tab structure (Request / Response / Errors / Raw) is preserved. The Errors and Raw tabs are NOT changed. All dynamic content continues to be HTML-escaped via `escapeHtml`. The implementation is **fully TypeScript** — the project is being migrated to TS, so `log-detail.ts` is the single source of truth and the compiled `dist/components/log-detail.js` is regenerated by tsc on every build. The hand-maintained `log-detail.js` is being deleted.

This spec is **frontend-only**. It also documents, but does NOT fix, a backend bug that explains why the Response tab is currently empty for streaming requests.

---

## Bug Investigation

### Symptom

In the admin dashboard's **Live Logs** view, clicking a row opens the detail modal. The **Response** tab shows `(empty)` (or just `(empty)` inside the `<pre class="json-viewer">`) for the vast majority of rows — i.e. streaming chat completions, which are the dominant case.

### Root cause (verified, in a separate work item)

The streaming-success path in `crates/openproxy-core/src/pipeline.rs:2877-2900` records the attempt via `record_attempt_raw_with_tokens(...)` with the response body hard-coded to `None`:

```rust
// crates/openproxy-core/src/pipeline.rs:2888-2900
let _ = self.record_attempt_raw_with_tokens(
    req, combo, target, Some(model), None,
    Some(connect_and_send_ms), ttft_ms, total_ms,
    status_code, attempt, race_size, trace_id,
    prompt_tokens, completion_tokens,
    None, // request_body_json
    None, // response_body_json      <-- always None on the streaming path
    None, // request_headers
    None, // response_headers
    true,        // is_streaming
    done_sent,   // stream_complete
    stop_reason,
);
```

The function signature in the same file at `pipeline.rs:3158-3159` confirms the parameter is `response_body_json: Option<serde_json::Value>`. The `usage.response_body_json` column exists in the schema (see `crates/openproxy-core/src/usage.rs:803` — `pub response_body_json: Option<Value>` on `UsageDetailRow`) and is read correctly by the modal in `log-detail.ts` via `log.response ?? detail["response"] ?? log.response_body_json ?? null`. The column is simply never written for streaming.

For non-streaming requests, the non-streaming dispatch path persists the buffered response and the column is populated, which is why the bug appears "for most rows" but not "for all rows."

### Consequence in the modal

In `log-detail.ts`:

```ts
const response: unknown = log.response ?? detail["response"] ?? log.response_body_json ?? null;
```

…evaluates to `null` for every streaming request, which `formatJson(null)` renders as the literal string `"(empty)"`.

### Proposed backend fix (OUT OF SCOPE for this spec; captured as a separate work item)

Two options, in order of preference:

1. **Capture the full SSE body on the streaming path** (mirror what the non-streaming path does for `final_response`): accumulate the parsed chat-completion response into a `serde_json::Value` as the chunks arrive (already partially done by per-executor parsers — e.g. `executor_antigravity.rs:407-434` and `executor_kiro.rs:288-325` maintain an `accumulated_text` / `usage` pair; extend that to also collect `reasoning_content` and `tool_calls`), then pass it as the `response_body_json` argument to `record_attempt_raw_with_tokens` instead of `None`. The schema column already supports it.
2. **Change the frontend to surface "body not captured for streaming"** as a permanent placeholder if option 1 is rejected for cost/storage reasons. This is the explicit placeholder this spec mandates below; it doubles as a useful diagnostic state until the backend is fixed.

### Impact on this frontend spec

Even with the body `null` for streaming, the modal must still:

- Render the body correctly when it IS present (non-streaming responses, and any future row that the backend fix populates).
- Show a clear, distinct placeholder when the body is `null` — not the current ambiguous `(empty)`.

This way, deploying only the frontend change does not regress existing behavior and gives operators a clear "we know we don't have this" signal instead of a silent blank.

## Response Tab Design

### Goals

- When the body is an OpenAI-style chat completion, surface the three things an operator actually looks at: **message**, **reasoning**, **tool calls**. Each in its own collapsible block.
- When the body is some other shape (Anthropic-style, embeddings, custom proxy response, etc.), show a single collapsible "Raw response" fallback so the data is still visible.
- When the body is `null`, show a clear placeholder that names the cause (streaming not recorded) and points to the backlog — so operators don't think the row is corrupt.

### Parsing rules

`renderResponseTab` does the string-to-JSON normalization before calling `parseOpenAiChatResponse`. Specifically, if `value` is a string, `renderResponseTab` attempts `JSON.parse` and recurses with the parsed value; on parse failure it formats the string as-is inside the "Raw response" collapsible. `parseOpenAiChatResponse` itself only ever receives a non-string value (or `null`/undefined for the empty state).

The modal reads the response value as `unknown`. The renderer classifies it before producing HTML:

- `null` or `undefined` → placeholder (see "Empty state" below).
- `string` → wrap in a single collapsible "Raw response" block with the string pretty-printed (JSON.parse first; if it parses, format the parsed value; if not, format the string itself).
- Anything else → coerce to an `object` and try to recognize an OpenAI chat completion shape:
  - Detect chat completion if **all** of the following hold:
    - value is a non-array object, AND
    - `Array.isArray(value.choices)` and `value.choices.length >= 1`, AND
    - `value.choices[0]` is a non-null object.
  - The message is read from `choices[0].message` (also accept `choices[0].text` as a fallback for the completion-style shape).
  - Reasoning is read, in order, from: `choices[0].message.reasoning_content` → `choices[0].message.reasoning` → `choices[0].message.reasoning_text`. Each candidate is considered present only if `typeof v === "string" && v.length > 0`. The first one matching wins; if none match, the Reasoning block is omitted.
  - Tool calls are read from `choices[0].message.tool_calls`. Must be an array of objects, each with `{ id?: string, type?: string, function: { name: string, arguments: string | object } }`. The `arguments` field is commonly a JSON-encoded string; try `JSON.parse(arguments)`, fall back to rendering the raw string.
- If the chat-completion detection does not hold, fall through to "Raw response" fallback (do NOT throw, do NOT show `{}`).

### Empty state (body is null)

Render exactly this in the Response section's body (no collapsibles, no JSON viewer):

```html
<section class="log-detail-section" data-log-tab="response">
  <h4>Response</h4>
  <p class="muted log-detail-placeholder">
    Response body not captured (streaming requests are not recorded yet — see backlog).
  </p>
</section>
```

The placeholder text is the value of the `STREAMING_PLACEHOLDER_TEXT` constant at the top of `log-detail.ts`:

```ts
const STREAMING_PLACEHOLDER_TEXT = "Response body not captured (streaming requests are not recorded yet — see backlog).";
```

The CSS class `log-detail-placeholder` is new and is defined in the CSS section below. It is visually distinct from `(empty)` so an operator can tell the difference between "we have it and it's literally empty" and "we don't have it at all."

### Filled state (chat completion recognized)

Render exactly this structure (one `<details>` per populated block; the message block is always shown if we recognized the chat shape):

```html
<section class="log-detail-section" data-log-tab="response">
  <h4>Response</h4>

  <details class="log-detail-collapsible" open>
    <summary>Message</summary>
    <pre class="json-viewer log-detail-collapsible-body">{escaped content text}</pre>
  </details>

  <details class="log-detail-collapsible" open>
    <summary>Reasoning</summary>
    <pre class="json-viewer log-detail-collapsible-body">{escaped reasoning text}</pre>
  </details>

  <details class="log-detail-collapsible" open>
    <summary>Tool calls (N)</summary>

    <div class="log-detail-tool-call">
      <div class="log-detail-tool-call-header">
        <span class="log-detail-tool-call-name">{escaped function name}</span>
        {optional: <span class="log-detail-tool-call-id">{escaped id}</span>}
      </div>
      <details class="log-detail-collapsible log-detail-tool-call-args">
        <summary>Arguments</summary>
        <pre class="json-viewer log-detail-collapsible-body">{escaped pretty JSON args}</pre>
      </details>
    </div>

    <!-- repeat log-detail-tool-call block for each tool call -->
  </details>
</section>
```

Rules:

- The `Message` block is omitted entirely if `content` (or `text`) is `null`, missing, or an empty string.
- The `Reasoning` block is omitted entirely if none of `reasoning_content` / `reasoning` / `reasoning_text` is a non-empty string (see "Parsing rules" for the exact predicate).
- The `Tool calls` block is omitted entirely if `tool_calls` is missing, not an array, or empty. If present, render one `log-detail-tool-call` block per element.
- The `Message` `<pre>` shows the string content escaped via `escapeHtml`. If the content is a JSON-encoded string (rare but possible), it is shown as-is — do not try to parse it again.
- The `Reasoning` `<pre>` is identical to the Message `<pre>` in styling and escaping rules.
- The `Tool calls` summary text is the literal `Tool calls (N)` where N is `tool_calls.length`. If N is 1, the text is still `Tool calls (1)` (do not pluralize).
- The Arguments `<details>` defaults to `open` for the first tool call in the row, and collapsed for subsequent ones — this matches how an operator reads them (top-down, expand as needed).
- The function name in the header is always shown. The `id` and `type` are shown only if present and are rendered with muted styling.

### `parseToolCallArguments` behavior

The function returns `{ pretty: string }` (no `parsed` field). It does NOT call `escapeHtml` itself — the caller wraps the result with `escapeHtml` (or feeds it to `formatJson` which already escapes).

Cases:

- `args === null` or `args === undefined` → the Arguments `<details>` block is NOT rendered at all. The tool-call header still shows the function name, but no Arguments subsection is produced.
- `typeof args === "string"` → try `JSON.parse`. On success: `{ pretty: JSON.stringify(parsed, null, 2) }`. On failure: `{ pretty: args }` (the raw string, unparsed).
- Array or object (and not null): `{ pretty: JSON.stringify(args, null, 2) }`.
- Primitive number/boolean: `{ pretty: String(args) }`.

The caller then applies `escapeHtml` to the `pretty` string before interpolating it into the `<pre>` body.

### Fallback state (recognized, but not chat-completion)

Render exactly:

```html
<section class="log-detail-section" data-log-tab="response">
  <h4>Response</h4>
  <details class="log-detail-collapsible" open>
    <summary>Raw response</summary>
    <pre class="json-viewer log-detail-collapsible-body">{escaped pretty JSON}</pre>
  </details>
</section>
```

This is the only state where the visible block says "Raw response" — it is intentionally distinct from the modal's top-level "Raw" tab so it cannot be confused with a full-row dump.

## Request Tab Design

### Goals

- When `request_body_json` is a JSON object, show each top-level key as a collapsible item, key name as the summary, value as pretty-printed JSON inside.
- Highlight the keys an operator reaches for first (`model`, `messages`, `tools`, `temperature`, `stream`, `max_tokens`).
- Handle non-object bodies (arrays, primitives) gracefully.
- Keep the existing "No request body recorded." placeholder.

### Parsing rules

- `null` or `undefined` → "No request body recorded." (unchanged from the current `hasRequestBody` check).
- non-empty object → per-key collapsibles, see "Filled state" below.
- empty object `{}` → placeholder (preserve the current `hasRequestBody` semantics; this currently treats `{}` as "no body").
- array or primitive → single collapsible "Raw request body" block with the full value pretty-printed.
- string that looks like JSON (`startsWith("{")` or `startsWith("[")`) → try `JSON.parse`; on success, treat as object/array per the above; on failure, fall through to the primitive case.
- A string that does NOT start with `{` or `[` is treated as a primitive and rendered verbatim inside the `Raw request body` collapsible via `formatJson`.

### Filled state (object body)

Render exactly:

```html
<section class="log-detail-section" data-log-tab="request">
  <h4>Request</h4>

  <details class="log-detail-collapsible" open>
    <summary><span class="log-detail-key log-detail-key-pinned">model</span></summary>
    <pre class="json-viewer log-detail-collapsible-body">"gpt-4o-mini"</pre>
  </details>

  <details class="log-detail-collapsible" open>
    <summary><span class="log-detail-key log-detail-key-pinned">messages</span> <span class="log-detail-key-meta">2 items</span></summary>
    <pre class="json-viewer log-detail-collapsible-body">[ ... pretty JSON ... ]</pre>
  </details>

  <details class="log-detail-collapsible">
    <summary><span class="log-detail-key log-detail-key-pinned">tools</span> <span class="log-detail-key-meta">1 item</span></summary>
    <pre class="json-viewer log-detail-collapsible-body">[ ... pretty JSON ... ]</pre>
  </details>

  <details class="log-detail-collapsible" open>
    <summary><span class="log-detail-key log-detail-key-pinned">stream</span></summary>
    <pre class="json-viewer log-detail-collapsible-body">true</pre>
  </details>

  <details class="log-detail-collapsible" open>
    <summary><span class="log-detail-key log-detail-key-pinned">temperature</span></summary>
    <pre class="json-viewer log-detail-collapsible-body">0.7</pre>
  </details>

  <details class="log-detail-collapsible" open>
    <summary><span class="log-detail-key log-detail-key-pinned">max_tokens</span></summary>
    <pre class="json-viewer log-detail-collapsible-body">1024</pre>
  </details>

  <!-- any other top-level key, in original object order, after the pinned keys -->
  <details class="log-detail-collapsible">
    <summary><span class="log-detail-key">user</span></summary>
    <pre class="json-viewer log-detail-collapsible-body">"u-123"</pre>
  </details>
</section>
```

Rules:

- **Pinned keys** (rendered first, in this order): `model`, `messages`, `tools`, `temperature`, `stream`, `max_tokens`. They get the `log-detail-key-pinned` class which gives them a subtle visual highlight (e.g. `color: var(--color-primary)`). Pinned keys that are not present in the object are simply skipped — no empty `<details>` for absent keys.
- **Remaining keys** are rendered in the original insertion order of the object, after the pinned keys, without the `pinned` class.
- **Default open state** (unambiguous): Pinned keys are open by default ONLY if their value is a primitive (string, number, boolean, null). Pinned keys with array or object values are closed by default. Non-pinned keys are closed by default regardless of value type. The example above follows this: `model`/`temperature`/`stream`/`max_tokens` are primitives and use `open`; `messages` and `tools` are arrays and use the closed form.
- **`log-detail-key-meta`** shows, in muted text, a short hint of the value shape for arrays and objects: `N items` for arrays (or `empty` if length 0), and `N keys` for objects (or `empty` if 0 keys). This lets the operator triage without expanding everything. Primitives get no meta hint.
  - Pluralization: arrays of length 1 show `1 item`, of length N (N>1) show `${N} items`. Objects of length 1 show `1 key`, of length N (N>1) show `${N} keys`. For length 0, the meta text is the literal string `empty` (no pluralization).
- Each value is rendered via `formatJson(value)` so the existing pretty-printer and `escapeHtml` are reused.
- Top-level keys are always shown as `escapeHtml(key)` — no special characters in keys are interpreted.

### Empty / non-object state

For `null`: keep the existing placeholder:

```html
<section class="log-detail-section" data-log-tab="request">
  <h4>Request</h4>
  <p class="muted">No request body recorded.</p>
</section>
```

For an empty object `{}` (and the existing `hasRequestBody` returns false for it), use the same placeholder — operators do not need to see "0 keys".

For an array or primitive, use the same fallback shape as the Response "Raw response" case:

```html
<section class="log-detail-section" data-log-tab="request">
  <h4>Request</h4>
  <details class="log-detail-collapsible" open>
    <summary>Raw request body</summary>
    <pre class="json-viewer log-detail-collapsible-body">{escaped pretty JSON}</pre>
  </details>
</section>
```

### Function placement (within `log-detail.ts`)

Place the new helpers (`parseOpenAiChatResponse`, `parseToolCallArguments`, `renderResponseTab`, `renderRequestTab`) and the `PINNED_REQUEST_KEYS` and `STREAMING_PLACEHOLDER_TEXT` constants in this order, immediately after `jsonSection` and before `renderLogDetailModal`. This keeps the public surface (`renderLogDetailModal`, `showLogDetail`, etc.) and the small helpers (`formatJson`, `jsonSection`, `statusPillClass`, `readString`) grouped at the top, and the new renderers next to each other in the middle.

### Constants (in `log-detail.ts`)

```ts
const STREAMING_PLACEHOLDER_TEXT =
  "Response body not captured (streaming requests are not recorded yet — see backlog).";

const PINNED_REQUEST_KEYS: readonly string[] = [
  "model", "messages", "tools", "temperature", "stream", "max_tokens",
];
```

`STREAMING_PLACEHOLDER_TEXT` is referenced from `renderResponseTab` whenever the body is `null` or `undefined`. `PINNED_REQUEST_KEYS` is referenced from `renderRequestTab` to order the per-key collapsibles. (No magic numbers; the list of pinned keys and the placeholder text are the only "magic" and both are named.)

### Functions to add (in `log-detail.ts`)

These are private helpers, no new exports.

- `parseOpenAiChatResponse(value: unknown): { message: string | null; reasoning: string | null; toolCalls: ToolCall[] } | null` — returns `null` if the value is not a recognized OpenAI chat-completion shape; otherwise returns the three extracted fields. See "Parsing rules" in the Response section for the exact precedence. The function expects a non-string value (the string normalization happens in `renderResponseTab`).
- `parseToolCallArguments(args: unknown): { pretty: string }` — see the dedicated "parseToolCallArguments behavior" subsection for the four cases (null/undefined, string, object/array, primitive number/boolean). Does NOT call `escapeHtml`; the caller does that.
- `renderResponseTab(response: unknown): string` — returns the full `<section data-log-tab="response">…</section>` HTML, picking the empty / chat-completion / fallback shape per the rules above. Performs the string-to-JSON normalization (see "Parsing rules") before calling `parseOpenAiChatResponse`. Uses `STREAMING_PLACEHOLDER_TEXT` for the empty state.
- `renderRequestTab(requestBody: unknown): string` — caller passes the already-resolved `requestBody` value (post `log.request_body_json ?? detail.request_body_json`). The function internally applies the `hasRequestBody` predicate to decide between the placeholder, the per-key collapsibles, and the "Raw request body" fallback. Returns the full `<section data-log-tab="request">…</section>` HTML.

### Functions to refactor

- `jsonSection(title, value, tabKey)` is **kept** for the Errors and Raw tabs (the current Response call site moves to `renderResponseTab`). No change to its signature or behavior.
- `renderLogDetailModal(log)` is refactored to call `renderRequestTab(requestBody)` and `renderResponseTab(response)` instead of inlining the section HTML for those two tabs. The Errors and Raw calls still go through `jsonSection(...)`.
- The `response: unknown = log.response ?? detail["response"] ?? log.response_body_json ?? null;` line is **unchanged** — it is the value passed into `renderResponseTab`.
- The `requestBody: unknown = log.request_body_json ?? detail.request_body_json;` line is **unchanged** — it is the value passed into `renderRequestTab`.

### BUG-INIT-1: `initializeLogDetailTabs` active-tab selector is wrong

**Current state.** `initializeLogDetailTabs` queries for `[data-log-detail-tab]` but the rendered HTML uses `data-action="logDetailTab"`. The selector matches nothing, so the first tab never gets `.active` set on initial render. This means clicking a row opens the modal with NO tab visually selected, and the operator sees the request-tab content (the default first section) without the "Request" tab being highlighted. The "first section is visible" effect comes only from the `data-log-tab` initial `style.display` set by the renderer — not from the `.active` class.

**Fix.** Change the selector inside `initializeLogDetailTabs` to find the first `.detail-tab` element within `.log-detail-tabs` and add `.active` to it. The new selector:

```ts
const firstTab = modal.querySelector(".log-detail-tabs .detail-tab");
if (firstTab) {
  firstTab.classList.add("active");
}
```

This single change covers both the initial-render path (called from `showLogDetail`) and the in-place re-render path (called from `updateOpenLogDetail`), because both call the same `initializeLogDetailTabs` function.

**Why this fix is correct.** The CSS rule `.detail-tab.active` already exists (it controls the active-tab visual state). The data-driven handler `logDetailTab` in `registry.ts` operates on the click event and toggles `.active` correctly. The bug is specifically that the initial population of the `.active` class is wrong — the selector is looking for an attribute that does not exist on the rendered tab buttons. Fixing the selector resolves it without touching the handler or the CSS.

**Acceptance.** AC-INIT-1 below codifies the test.

### BUG-STACK-1: modal stacking when `showLogDetail` is called twice

**Current state.** If `showLogDetail` is called twice in a row without an intervening `closeLogDetailModal`, two `.log-detail-modal` elements end up in the DOM stacked on top of each other. Click handling, focus, and the close button all start to misbehave because there are now two backdrops and two modal cards.

**Fix.** At the start of `showLogDetail`, before appending the new modal, check for an existing `.log-detail-modal` in the DOM and remove it. One-liner:

```ts
document.querySelector(".log-detail-modal")?.remove();
```

This is a defensive guard; the legitimate use case is `updateOpenLogDetail` (which uses `outerHTML` replacement and does not stack), so the only way to reach the stacked state today is a programming error or a race — but guarding against it costs nothing and prevents confusing UX.

**Acceptance.** AC-STACK-1 below codifies the test.

### Functions to keep UNCHANGED (modulo the BUG-INIT-1 fix)

These preserve their existing public behavior; only `initializeLogDetailTabs` is touched (selector fix above):

- `statusPillClass`
- `formatJson`
- `jsonSection`
- `readString`
- `wireTabClickOnce`
- `tabClickOnce`
- `showLogDetail` (gets the BUG-STACK-1 one-liner prepended)
- `closeLogDetailModal`

- `hasCompleteLogDetail`

The `LogDetailLog` interface at the top of `log-detail.ts` is also unchanged.

### HTML escaping (security)

Every dynamic value in the new HTML — function names, tool-call ids, key names, message text, reasoning text, JSON pretty output, and the `(N items)` / `(N keys)` meta counts — must go through `escapeHtml`. The exception is JSON pretty output that already comes from `formatJson` (which already escapes). When a tool call's `arguments` is a JSON-encoded string, `JSON.parse` it BEFORE passing to `formatJson` so the output is properly escaped. The `pretty` field returned by `parseToolCallArguments` is also wrapped in `escapeHtml` by the caller.

### Tab switching compatibility

The `data-log-tab="response"` and `data-log-tab="request"` attributes on the outer `<section>` are preserved. The `logDetailTab` handler in `handlers/registry.ts:224-229` (which toggles `style.display` on `[data-log-tab]` sections) continues to work without modification. `initializeLogDetailTabs` (with the BUG-INIT-1 fix) re-walks the new sections and resets their `style.display` and `.active` class.

### In-place re-render compatibility

`updateOpenLogDetail` calls `modal.outerHTML = html` and then `wireTabClickOnce()` + `initializeLogDetailTabs()`. After this refactor, the new HTML still has the `id="log-detail-modal"` and `class="log-detail-modal"` (it is still the outer `<div>` of `renderLogDetailModal`), so `document.querySelector(".log-detail-modal")` in `updateOpenLogDetail` still finds the new element by class on the next call. `wireTabClickOnce` is idempotent. `initializeLogDetailTabs` re-walks the new sections and resets their `style.display` and `.active` class (BUG-INIT-1 fix). No changes needed in `updateOpenLogDetail`.
